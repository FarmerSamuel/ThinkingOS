# Right Problem (ThinkingOS)

Validates whether a stated problem is the problem that should be solved before solution generation begins.

**Thinking habit:** Validate the problem frame before generating or selecting solutions.

Generated from the canonical package `skills/right-problem/` (version 1.0.1). Do not edit this file; update the package and run `python tools/export_claude_skills.py`.

## Purpose

People often begin solving an ambiguous, symptom-led, or solution-led problem before validating the goal, obstacle, constraints, and supporting evidence.

Target outcome: A clear, bounded, evidence-aware problem frame with an explicit validity status and next thinking step.

### In scope

- Clarifying the desired outcome.
- Separating goals, obstacles, constraints, assumptions, and evidence.
- Testing whether the obstacle plausibly explains the goal gap.
- Identifying missing information and alternative explanations.
- Evaluating problem-frame readiness.

### Out of scope

- Generating solutions.
- Building implementation plans.
- Diagnosing people or organizations.
- Replacing domain, legal, medical, or financial expertise.

## Position in the thinking graph

This is an entry-point skill with no prerequisite skills.

- Consumes: `ProblemStatement`, `Goal`, `Context`.
- Produces: `ValidatedProblem`, `ClarifiedGoal`, `ProblemAssumptions`.

## Required inputs

| Input | Required | Description |
| --- | --- | --- |
| `problemStatement` | Yes | The problem as currently understood by the user. |
| `goal` | Yes | The desired outcome that makes the problem worth solving. |
| `context` | No | Relevant environment, stakeholders, evidence, history, and boundaries. |
| `knownConstraints` | No | Known boundaries governing the feasible problem space. |

Collect required inputs before analysis. When a required input is missing or materially ambiguous, ask the single highest-impact clarification question and wait for the answer.

## Execution pipeline

Follow the shared ThinkingOS Engine order:

```text
Collect Inputs -> Validate -> Detect Missing Information -> Evaluate Elements
-> Run Logic Rules -> Evaluate Reasoning -> Generate Evaluation
-> Generate Output -> Recommend Next Skill
```

A later stage must not bypass an unresolved blocking result from an earlier stage.

## Validation gates

- **RP-V01** — A non-empty problem statement must be provided. If it fails: Return needs clarification and request the observed undesirable condition or decision difficulty.
- **RP-V02** — A desired outcome must be distinguishable from an activity or preselected solution. If it fails: Return needs clarification and ask what outcome should improve.
- **RP-V03** — Materially ambiguous scope, stakeholder, timeframe, or terminology must be clarified. If it fails: Ask the single highest-impact clarification question.
- **RP-V04** — Goals, obstacles, constraints, assumptions, and evidence must be internally consistent. If it fails: Record the contradiction and classify the frame as incomplete or invalid.
- **RP-V05** — Success criteria must be observable enough to distinguish improvement from activity completion. If it fails: Request the smallest success definition that would materially change evaluation.

## Logic rules

Apply in order:

1. **RP-L01** — Separate the desired outcome from the current problem statement and any proposed solution. If it fails: Do not proceed to solution generation.
2. **RP-L02** — Identify the obstacle claimed to explain the gap between the current condition and goal. If it fails: Request the observed factor preventing the goal.
3. **RP-L03** — Test the obstacle against evidence, competing explanations, and symptom-versus-cause confusion. If it fails: Lower Logic Score and request discriminating evidence.
4. **RP-L04** — Separate governing constraints from obstacles and unsupported assumptions. If it fails: Lower Constraint Score and disclose the classification gap.
5. **RP-L05** — Classify the complete frame as Valid, Invalid, or Incomplete using blocking conditions and rubric evidence. If it fails: Return Incomplete with the unresolved inconsistency.
6. **RP-L06** — Challenge the frame with at least one plausible alternative explanation or disconfirming observation when causality is material. If it fails: Lower confidence and request evidence that discriminates between explanations.

## Conversation rules

- Reflect the current problem and goal before challenging the frame.
- Ask one decision-relevant question at a time.
- Prioritize the unknown most likely to change problem validity.
- Use neutral Socratic questions and avoid leading the user.
- State assumptions before using them.
- Stop questioning when the frame is sufficient for the current decision.

## Evaluation

Score each criterion from 0 to 4:

| Criterion | Weight | Description |
| --- | --- | --- |
| Goal Score | 0.25 | Clarity, relevance, scope, and observability of the desired outcome. |
| Obstacle Score | 0.25 | Specificity, evidence, and causal plausibility of the identified obstacle. |
| Constraint Score | 0.2 | Completeness and correct classification of material boundaries. |
| Logic Score | 0.3 | Consistency and evidential support connecting the goal, obstacle, constraints, and conclusion. |
| Overall Thinking Score | — | Problem-frame readiness after applying blocking safeguards. |

Scale: 0 = absent-or-invalid, 1 = materially-weak, 2 = partially-usable, 3 = sufficient, 4 = decision-ready.

## Output format

Return one fenced JSON object that conforms to the ThinkingOS universal
output contract (`schemas/output.schema.json`), then a short plain-language
recap of the recommendation and next step.

Required fields:

- `summary` — one-paragraph result including the frame status.
- `evaluation` — object with `score`, `strengths`, `weaknesses`,
  `missingInformation`, `logicalIssues`, `recommendation`, `confidence`.
- `reasoning` — array of strings; report each rubric dimension as a structured
  entry (for example `Goal Score: 3/4 — supported by stated evidence`).
- `assumptions` — every assumption the analysis relied on.
- `missingInformation` — unknowns that could materially change the result.
- `questions` — outstanding clarification questions (at most one prioritized
  question when a blocking gap exists).
- `nextStep` — the single most useful next action.
- `confidence` — `low`, `medium`, or `high` (or a number from 0 to 1).

Preserve material uncertainty. Never hide a blocking weakness behind a strong
overall score, and never treat fluent prose as evidence of valid reasoning.

## Never rules

- Never generate solutions before validating the problem frame.
- Never invent goals, obstacles, constraints, stakeholders, or evidence.
- Never treat a proposed solution as a validated problem.
- Never attribute motive, character, or capability without evidence.
- Never hide a blocking ambiguity behind an overall score.
- Never recommend a downstream skill whose prerequisites are unsatisfied.
- Never treat an ambiguity checklist, Five Whys chain, or resource test as proof.

## Next skill transitions

Recommend at most one downstream skill:

- `constraints` — Condition: The frame is valid and governing constraints require explicit analysis. Reason: Define the feasible problem space before evaluating options.
- `break-it-down` — Condition: The frame is valid but too broad or complex for direct analysis. Reason: Decompose the validated problem into tractable components.
- `gap-analysis` — Condition: The frame is valid and initial and target states are sufficiently defined. Reason: Identify and prioritize the differences between current and desired states.

Never recommend a downstream skill whose prerequisites are unsatisfied. When no transition condition holds, recommend a direct action or clarification instead.
