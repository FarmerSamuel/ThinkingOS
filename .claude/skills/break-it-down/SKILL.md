# Break It Down (ThinkingOS)

Decomposes a validated problem into coherent, dependency-aware minimum tractable units.

**Thinking habit:** Make a complex problem tractable by decomposing it without losing meaning or dependencies.

Generated from the canonical package `skills/break-it-down/` (version 1.0.0). Do not edit this file; update the package and run `python tools/export_claude_skills.py`.

## Purpose

Broad or coupled problems cannot be evaluated reliably as a single undifferentiated whole.

Target outcome: A goal-aligned component structure with tractable leaves and explicit dependencies.

### In scope

- Selecting a decomposition basis.
- Creating components.
- Testing coverage and overlap.
- Mapping dependencies.
- Defining tractable units.

### Out of scope

- Validating the original problem.
- Solving components.
- Assigning implementation work.
- Selecting technology.

A later stage must not bypass an unresolved blocking result from an earlier stage.

## Validation gates

- **BID-V01** — The problem and goal must be present and consistent. If it fails: Request prerequisite clarification.
- **BID-V02** — The decomposition basis must be coherent at each level. If it fails: Reject mixed levels and select one basis.
- **BID-V03** — The problem boundary must be sufficiently clear. If it fails: Request boundary clarification.

## Logic rules

Apply in order:

1. **BID-L01** — Generate candidate components using one declared basis. If it fails: Refine the basis or component level.
2. **BID-L02** — Test collective coverage and sibling overlap. If it fails: Add missing components or resolve overlap.
3. **BID-L03** — Refine leaves until stopping criteria are met. If it fails: Continue decomposition or disclose the limit.
4. **BID-L04** — Preserve dependencies and cross-cutting concerns. If it fails: Record dependency issues and reduce confidence.

## Conversation rules

- Confirm the decomposition basis before expanding the tree.
- Ask one boundary or structure question at a time.
- Explain why further decomposition is or is not useful.
- Do not turn components into premature solutions.

## Evaluation

Score each criterion from 0 to 4:

| Criterion | Weight | Description |
| --- | --- | --- |
| Coverage Score | 0.25 | Completeness of in-scope representation. |
| Distinctness Score | 0.15 | Separation of sibling responsibilities. |
| Tractability Score | 0.2 | Fitness of leaves for reliable analysis. |
| Dependency Score | 0.2 | Preservation of sequencing and coupling. |
| Goal Alignment Score | 0.2 | Traceability of components to the validated goal. |

Scale: 0 = unusable, 1 = weak, 2 = partial, 3 = sufficient, 4 = robust.

## Output format

Return one fenced JSON object that conforms to the ThinkingOS universal
output contract (`schemas/output.schema.json`), then a short plain-language
recap of the recommendation and next step.

Required fields:

- `summary` — one-paragraph result including the frame status.
- `evaluation` — object with `score`, `strengths`, `weaknesses`,
  `missingInformation`, `logicalIssues`, `recommendation`, `confidence`.
- `reasoning` — array of strings; report each rubric dimension as a structured
  entry (for example `Goal Score: 3/4 — supported by stated evidence`).
- `assumptions` — every assumption the analysis relied on.
- `missingInformation` — unknowns that could materially change the result.
- `questions` — outstanding clarification questions (at most one prioritized
  question when a blocking gap exists).
- `nextStep` — the single most useful next action.
- `confidence` — `low`, `medium`, or `high` (or a number from 0 to 1).

Preserve material uncertainty. Never hide a blocking weakness behind a strong
overall score, and never treat fluent prose as evidence of valid reasoning.

## Never rules

- Never run without a validated problem and goal.
- Never mix decomposition bases silently.
- Never hide overlap, omissions, or dependencies.
- Never equate more detail with better decomposition.
- Never solve components during decomposition.
