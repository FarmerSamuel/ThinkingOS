# ThinkingOS Glossary

This glossary defines shared, platform-independent terms. Skill-specific definitions may narrow a term but must not contradict it without an explicit versioned extension.

| Term | Definition |
| --- | --- |
| Assumption | A claim provisionally accepted without sufficient validation. |
| Boundary | A declared distinction between what is inside, outside, or crossing a system under analysis. |
| Confidence | Calibrated reliability of an evaluation given evidence quality, completeness, and uncertainty. |
| Constraint | A condition that limits the feasible problem or decision space; it may be fixed, negotiable, assumed, or unknown. |
| Context | Environmental information that changes the meaning, relevance, or feasibility of reasoning. |
| Dependency | A prerequisite capability, input, or relationship required before another operation can run reliably. |
| Element | A typed unit of thought examined by a skill, represented as `$Element` in ThinkingOS Language. |
| Evaluation | A structured assessment of thinking quality, limitations, recommendation, and confidence. |
| Evidence | Information relevant to supporting, challenging, or discriminating between claims. |
| Goal | A desired outcome used to evaluate relevance and success. |
| Heuristic | A bounded decision shortcut with declared applicability and failure conditions. |
| Logic rule | An explicit, testable relationship or operation applied to validated thinking elements. |
| Obstacle | A condition that plausibly explains why a goal is not currently achieved. |
| Problem frame | A structured representation of goal, current condition, obstacle, constraints, evidence, assumptions, and scope. |
| Skill | A versioned ThinkingOS capability that applies one reusable thinking habit to one thinking problem. |
| State | Explicit reusable context describing the active skill, goal, inputs, pending questions, and next skill. |
| Validated input | Input that passed presence, relevance, ambiguity, consistency, and dependency checks, possibly with disclosed assumptions. |

## Naming authority

DSL naming follows [ThinkingOS Language](../docs/thinking-language.md); machine fields follow the project schemas. New glossary entries require review for overlap and compatibility.
