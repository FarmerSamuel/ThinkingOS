# Problem-Solving Knowledge

## Problem before solution

A problem-solving process begins with a validated outcome gap, not a preferred intervention. The problem frame should separate:

- Desired outcome.
- Current condition.
- Plausible obstacle or causal hypothesis.
- Governing constraints.
- Evidence and assumptions.
- Scope, stakeholders, and success criteria.

## Structured progression

```text
Frame → Validate → Decompose → Analyze → Decide → Test → Learn
```

Each transition has a gate. Missing evidence or contradiction should return the process to the earliest affected stage rather than be carried forward as false certainty.

## Decomposition

Choose one organizing basis per level. Test collective coverage, sibling overlap, tractability, goal alignment, and hidden dependencies. The smallest unit is not always the best unit; stop when further division reduces explanatory or decision value.

## From analysis to action

Recommendations should state the evidence, logic, assumptions, uncertainty, trade-offs, owner, and next validation step. Prefer reversible experiments when uncertainty is material and feedback is available.

## Common failure modes

- Solving symptoms while preserving their cause.
- Bundling unrelated problems.
- Assuming every constraint is fixed.
- Decomposing away cross-cutting dependencies.
- Declaring completion without a success measure.

## Skill links

The typical foundation is `#RightProblem → #Constraints → #BreakItDown`, followed by a goal-specific analysis skill rather than an automatic fixed sequence.
