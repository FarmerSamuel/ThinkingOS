# ThinkingOS 架構

## 概觀

ThinkingOS 採用分層架構，使推理行為可重複使用、可測試，且不依賴 AI 供應商或操作介面。

```text
Core
  ↓ Engine
  ↓ Specification
  ↓ Skill
  ↓ Knowledge
  ↓ Adapters
```

箭頭表示相依方向：下層使用上層建立的契約。跨層存取只能透過明確介面與 Schema。

## 各層責任

### Core

定義所有實作共用的角色、哲學、原則、驗證要求、邏輯契約、輸出格式、工作流程、提問方式與對話狀態。Core 不得包含特定技能方法、領域知識或平台整合。

### Engine

執行共用推理管線，協調輸入收集、驗證、缺漏偵測、元素評估、邏輯規則、推理評估、輸出產生與下一技能建議。Engine 控制順序，但不創造技能規則或領域事實。

### Specification

定義技能、評估、輸出及 ThinkingOS Language 的正式契約。文件與 Schema 讓不同實作都能接受一致性測試；它描述有效元件必須具備什麼，而非特定 Runtime 如何實作。

### Skill

每個技能封裝一種思考習慣與一個思考問題，宣告輸入、驗證、邏輯、評估、對話、輸出、測試與允許的轉換。技能透過共用狀態與 Schema 組合，不得重複 Core 或 Engine 的責任。

### Knowledge

提供技能可引用的證據、概念、分類與參考資料。知識必須有版本與來源，並與邏輯分離，使兩者能獨立演進。知識不能略過驗證，也不能在沒有適用規則時直接決定結論。

### Adapters

連接 LLM、API、儲存、介面與工具，在平台格式和共用契約之間轉換。憑證、模型設定、傳輸細節與介面行為只能存在這一層。

## 架構限制

- 相依關係必須經過文件化契約，平台細節不得向上滲漏。
- Core 與 Specification 必須保持 AI 中立。
- Engine 執行技能宣告的行為，不內建領域邏輯。
- 技能引用知識，而非複製知識。
- Adapter 可替換，但不能改變技能語意。
- 每一層都能獨立版本化與測試。
