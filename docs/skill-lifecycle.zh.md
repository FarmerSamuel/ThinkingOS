# ThinkingOS 技能生命週期

## 生命週期

```text
草稿 → 審查 → 測試 → 發布 → 改善 → 棄用
```

### 草稿

依技能規格定義單一思考習慣、目的、邊界、輸入、規則、評估、輸出、範例與測試。草稿不穩定，不能宣稱為已發布能力。

**退出條件：** 定義完整到足以進行 Schema 驗證與獨立審查。

### 審查

檢查概念清晰度、責任邊界、與現有技能的重疊、平台中立性、證據品質及 Core 一致性。

**退出條件：** 審查意見已解決或明確接受，且核准進入一致性測試。

### 測試

執行 Schema、正向、無效輸入、歧義、邊界、Never Rule、邏輯、輸出及跨平台一致性測試。

**退出條件：** 必要測試通過，已知限制已記錄。

### 發布

指定語意化版本，發布定義與測試，記錄變更並宣告相容性。已發布成品不可變更；修正必須用新版本。

**退出條件：** 成品可探索、有版本、有文件且可重現。

### 改善

收集實際使用、缺陷、評估品質與相容性證據。提案重新經過草稿、審查與測試。

### 棄用

技能不安全、重複、已被取代或無法維護時，標記為不建議使用，並提供原因、替代方案、支援期間及移除計畫。

## 語意化版本

- **MAJOR：** 輸入、驗證、邏輯、評估、輸出、狀態轉換或公開識別的不相容變更。
- **MINOR：** 向下相容的新能力、選填輸入、評估項目、範例或輸出擴充。
- **PATCH：** 不改變預期語意的修正、澄清、參考更新或測試改善。

預發布版可使用 `1.0.0-alpha.1`。`0.y.z` 代表初期開發，但仍須記錄破壞性變更。

## 相容性與變更控制

- 每次發布都要有 Changelog。
- 測試與範例跟著技能版本化。
- 使用者應固定相容版本範圍。
- 棄用新增生命週期資訊，不修改既有發布。
- 重新命名或拆分技能通常是破壞性變更，除非有明確且暫時的相容別名。
