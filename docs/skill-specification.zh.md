# ThinkingOS 技能規格

## 狀態與範圍

本文件定義每個 ThinkingOS Skill 都必須遵循的平台中立契約。技能是培養或套用一種思考習慣的可重複使用能力，不是模型提示詞、使用者介面、供應商整合或領域答案集合。

合規技能必須定義以下每一節；沒有適用內容時可明確寫「不適用」，但不得靜默省略。

## 標準技能結構

### 1. Metadata

提供穩定識別與生命週期資訊，包括名稱、版本、描述、狀態、作者、標籤與相容性，用於探索、版本管理、驗證與組合。

### 2. Purpose

定義技能處理的單一思考問題、預期成果與範圍邊界，說明何時應使用及不應使用。

### 3. Thinking Habit

命名技能所培養或套用的可重複認知行為。一個技能只能有一個主要思考習慣。

### 4. Thinking Elements

列出技能檢查的思考單元，例如主張、假設、證據、限制、替代方案、標準或後果；它描述思考對象，而非平台儲存方式。

### 5. Required Inputs

宣告執行所需資訊，包括型別、意義、必要性與限制。輸入不得依賴特定 LLM 訊息格式。

### 6. Validation Rules

定義分析前的存在性、格式、相關性、歧義、充分性與一致性檢查。每條規則要說明失敗條件，以及是否阻擋評估或允許已揭露假設。

### 7. Logic Rules

定義有順序、可測試的推理操作。規則必須足以讓相容實作重現，不能依賴隱藏模型行為。

### 8. Evaluation Criteria

定義如何依可觀察證據與共用評估模型判斷思考品質，而非獎勵使用者同意系統。

### 9. Conversation Rules

定義問題順序、澄清門檻、每回合資訊量與暫停時機。問題必須中立且只詢問會實質改善評估的內容。

### 10. Output Schema

宣告結構化結果。輸出必須符合 `schemas/output.schema.json`；特定技能擴充必須有文件且保留通用欄位。

### 11. Never Rules

列出技能特有的禁止行為與無效結論，形成可執行的安全與範圍邊界。

### 12. Next Skill

定義可選的後續技能、轉換條件與推進目前目標的理由。不得只為延長互動而呼叫下一技能。

### 13. References

列出支援技能的標準、研究、框架或內部規格，並提供足以區分外部證據與專案慣例的來源資訊。

### 14. Examples

提供平台中立的代表輸入與預期輸出案例；範例用於說明契約，不能取代規則與測試。

### 15. Tests

定義有效與無效輸入、歧義、邊界、邏輯、輸出結構及 Never Rules 的一致性案例；契約允許時應保持決定性。

## 合規要求

合規技能必須：

1. 通過 `schemas/skill.schema.json` 驗證。
2. 遵循 Core 驗證與 Logic Engine 契約。
3. 產生相容於 `schemas/output.schema.json` 的輸出。
4. 評估使用者思考時使用共用評估模型與 `schemas/evaluation.schema.json`。
5. 將 Adapter、模型設定與介面行為留在技能之外。
6. 明確宣告擴充，並在同一主版本內維持向下相容。
