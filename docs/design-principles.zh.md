# ThinkingOS 技能設計原則

## 一個技能，一種思考習慣

每個技能只培養或套用一種主要思考習慣。清楚的認知單元更容易理解、測試與重複使用。

## 一個技能，一個思考問題

技能只解決一個定義明確的思考問題。若規格同時需要不相關的目標或評估方式，應拆成可組合的技能。

## 以契約組合

技能透過共用 Schema 交換已驗證的狀態與結構化輸出，並使用明確轉換條件，不依賴隱藏的編排假設。

## 責任不重疊

每個技能擁有獨立目的與決策邊界；共同行為應放在 Core 或共用 Schema，而非複製到多個技能。

## 邏輯與知識可重複使用

邏輯規則應明確、盡可能具決定性，且不依賴呈現方式。知識資產應被引用而非重複嵌入，使邏輯和知識可以分別演進。

## 平台中立

技能不得依賴特定 LLM、供應商、API、訊息格式、工具協定或使用者介面；平台行為屬於 Adapter。

## 先驗證，再評估

任何技能在評估或建議前，都必須檢查輸入是否充分、是否有歧義及邏輯是否一致。未知資訊必須明確呈現，不得暗自補完。

## 推理透明且可測試

輸出要區分事實、假設、評估、推理與信心程度。權威感、流暢度或語氣確定性不能取代證據；每條規則與邊界都應能形成一致性測試。
